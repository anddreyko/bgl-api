# Domain-Driven Design (DDD)

## Date: 2024-01-15

## Authors: BoardGameLog Team

## Status: Accepted

---

### Context

BoardGameLog works with a rich domain model of board games, where specific business rules, entity relationships, and
complex aggregates exist. We needed to choose a modeling approach that adequately reflects the reality of the subject
area.

**Challenges:**

- Complex relationships between entities (games, sessions, players, statistics)
- Specific business rules (player count limits, result types)
- Need for a common language between developers and domain experts
- Integration with an external system (BGG) with its own data model

**Requirements:**

- Model must reflect real business concepts
- Business rules must be encapsulated in domain objects
- Code must use terminology understandable to domain experts

### Considered Options

#### Option 1: Anemic Domain Model (DTO + Services)

Data is stored in simple objects (DTOs), all logic is moved to services.

**Pros:**

- Simple object structure
- Easy to serialize
- Understandable for developers without DDD experience

**Cons:**

- Business logic is spread across services
- Difficult to guarantee invariants
- No protection against invalid object states
- Validation duplication in different places

#### Option 2: Domain-Driven Design

Rich domain model with aggregates, entities, Value Objects, and domain services.

**Pros:**

- Business rules are encapsulated in objects
- Ubiquitous Language between code and business
- Invariant protection at aggregate level
- Clear transactional consistency boundaries

**Cons:**

- High entry barrier for developers
- Requires deep domain understanding
- Risk of over-engineering for simple CRUD operations
- More code to support the model

#### Option 3: Active Record Pattern

Objects combining data and DB methods (like in Eloquent/Yii).

**Pros:**

- Fast CRUD functionality development
- Less boilerplate
- Intuitive

**Cons:**

- Strong coupling with database
- Difficult to test without DB
- Model is tied to table structure
- Business logic mixes with persistence

### Decision

**Decision:** Domain-Driven Design adopted (Option 2)

**Reason for choice:**

1. **Rich domain** — board games have many specific rules that are naturally expressed through DDD
2. **Ubiquitous Language** — terms Play, Game, Player, Mate are used both in code and communication
3. **Invariants** — critically important to guarantee data validity (e.g., player count in a session)
4. **Clean Architecture integration** — DDD pairs excellently with the chosen architectural approach

### Consequences

**Positive:**

- Business rules are protected at domain object level
- Code uses domain terminology understandable to stakeholders
- Clear aggregate boundaries simplify transaction understanding
- Value Objects guarantee base type validity (Email, PlayerId)

**Negative/Risks:**

- Team training time required for DDD concepts
- For simple CRUD operations, DDD may be excessive
- Constant model refactoring needed as domain understanding changes
- Complexity of mapping aggregates to relational model

### Notes

**Bounded Contexts in BoardGameLog:**

1. **Auth Context** — authentication and authorization
2. **Games Context** — board game catalog
3. **Plays Context** — game session logging
4. **Stats Context** — analytics and statistics
5. **Sync Context** — external source synchronization

**Aggregates:**

```
Play (Aggregate Root)
├── Player (Entity)
├── PlayId (VO)
├── GameId (VO)
└── Location (VO)

Game (Aggregate Root)
├── GameId (VO)
├── BggId (VO)
├── GameName (VO)
└── Expansion → Game

User (Aggregate Root)
├── UserId (VO)
├── Email (VO)
└── Mate (Entity)
```

**Value Object Rules:**

- Immutable
- Validated in constructor
- Compared by value
- Methods return new instances

**Entity Rules:**

- Have unique identifier
- Can change state through methods
- Compared by identifier
- Protect their invariants
