# Event-Driven Architecture: Development Strategy

## Date: 2024-01-25

## Authors: BoardGameLog Team

## Status: Accepted

---

### Context

When designing BoardGameLog, a question arises about using event-driven architecture. Several related but fundamentally
different concepts exist: domain events, Event Sourcing, integration events. It's important to clearly distinguish these
concepts and define an implementation strategy considering limited MVP resources.

**Key Constraint:** At the MVP stage, there are no resources for storing all events and supporting complex Event
Sourcing infrastructure.

**Requirements:**

- Loose coupling between system modules
- Ability to react to domain changes
- Development path to more complex patterns in the future
- Minimal MVP overhead
- Using existing MessageBus infrastructure

---

### Key Concepts

Before considering options, it's necessary to clearly distinguish three different concepts.

#### Domain Events

Events recording the fact of a significant change within the domain (e.g., `PlayCreated`, `PlayerAdded`). Their purpose
is to inform other parts of the system about changes for reaction.

**Characteristics:**

- Generated by aggregates after successful changes
- Implement `Core\Messages\Event` contract
- Distributed through `MessageBus` (same mechanism as for Command and Query)
- Generally **not stored** in DB — only processed within transaction
- Mechanism for decomposition and loose coupling within domain

#### Event Sourcing

A separate architectural pattern where aggregate state is **reconstructed** from the sequence of all occurred events.
Events are stored in Event Store as the single source of truth.

**Characteristics:**

- All events are saved in strict order
- Current state is obtained by applying all events in order
- Events contain all data to reconstruct state at any moment
- Requires significant infrastructure and storage resources

**Important:** Domain events can exist without Event Sourcing, but not vice versa.

#### Integration Events

Events for technical communication between different modules or microservices. They publish consistent facts already
committed in the system.

**Characteristics:**

- Not published by aggregates directly
- Sent to message broker
- Processed by background jobs
- Ensure consistency in distributed system

---

### Considered Options

#### Option 1: Event Sourcing from Day One

Full Event Sourcing with Event Store for all aggregates.

**Pros:**

- Complete audit of all changes
- "Time travel" capability
- State reproducibility

**Cons:**

- Significant storage resources
- Complex infrastructure
- Excessive for MVP
- High entry barrier for team

#### Option 2: Domain Events Only (Without Storage)

Domain events are generated and processed within transaction but not stored.

**Pros:**

- Minimal overhead
- Loose coupling between modules
- Simple implementation
- Sufficient for MVP
- Uses existing MessageBus

**Cons:**

- No event history
- No ability to reproduce state
- Events are lost on failure

#### Option 3: Outbox + Jobs for Integration

Domain events within process + Outbox pattern for reliable async communication.

**Pros:**

- Reliable event delivery
- Atomicity with business data
- Ready for distributed system

**Cons:**

- Additional Outbox table
- Background process for processing
- Excessive for monolith at MVP

---

### Decision

**Decision:** Phased event-driven architecture implementation using existing MessageBus

#### Phase 1: MVP

Only **domain events without storage**. Events implement `Core\Messages\Event` contract and are processed through
existing `MessageBus` synchronously within transaction.

```php
// Event implements existing contract
final readonly class PlayCreated implements Event
{
    public function __construct(
        public PlayId $playId,
        public GameId $gameId,
        public DateTimeImmutable $occurredAt,
    ) {}
}

// Aggregate generates event
$play = Play::create($gameId, $date);
$play->recordEvent(new PlayCreated($play->id(), $gameId, $date));

// MessageBus processes events (same mechanism as for Command/Query)
foreach ($play->releaseEvents() as $event) {
    $messageBus->handle($event);
}
```

**Application:**

- Updating related data (e.g., counters)
- Sending notifications (email, push)
- Cache invalidation

#### Phase 2: Expansion

Adding **Outbox pattern** for reliable async processing of critical events (BGG synchronization).

```
┌─────────────────────────────────────────────────────┐
│                   Transaction                       │
│  ┌──────────────┐    ┌──────────────┐               │
│  │ Play Table   │    │ Outbox Table │               │
│  │ INSERT play  │    │ INSERT event │               │
│  └──────────────┘    └──────────────┘               │
└─────────────────────────────────────────────────────┘
                           │
                           ▼
              ┌────────────────────────┐
              │   Background Worker    │
              │   (processes Outbox)   │
              └────────────────────────┘
                           │
                           ▼
              ┌────────────────────────┐
              │      BGG Sync Job      │
              └────────────────────────┘
```

#### Phase 3: Scaling (If Needed)

Considering **Event Sourcing** for specific aggregates with high audit requirements (e.g., session change history for
dispute resolution).

**Reason for choice:**

1. **Pragmatism** — MVP doesn't require full Event Sourcing
2. **Consistency** — events use same MessageBus as Command and Query
3. **Gradualness** — each phase adds capabilities without rewriting
4. **Resources** — minimal costs at start
5. **Flexibility** — can implement ES selectively for individual aggregates

---

### Consequences

**Positive:**

- Minimal MVP overhead
- Unified processing mechanism for all message types (Command, Query, Event)
- Clear concept separation in team
- Ready for scaling without architecture rewrite
- Domain events provide loose coupling already at MVP

**Negative/Risks:**

- No full event history at MVP
- Synchronous events may be lost on failure
- Discipline required: don't confuse domain events with Event Sourcing

---

### Notes

#### What Does NOT Replace Each Other

| Concept         | Purpose                       | Storage         |
|-----------------|-------------------------------|-----------------|
| **Event Store** | Business data (what happened) | Permanent       |
| **Outbox**      | Technical buffer for delivery | Temporary       |
| **Logs**        | System operation diagnostics  | Separate system |

**Event Store and Outbox don't replace logging.** Specialized systems are used for logs: ELK stack, Graylog, Sentry.

#### Integration with Existing Contracts

Domain events use existing Messages infrastructure:

```php
// src/Core/Messages/Event.php — already exists
interface Event extends Message {}

// src/Core/Messages/MessageBus.php — already exists
interface MessageBus
{
    public function handle(Message $message): mixed;
}
```

An event is just another Message type processed by the same MessageBus:

```php
// Message contracts (already exist)
interface Message {}
interface Command extends Message {}  // Changes state
interface Query extends Message {}    // Reads state
interface Event extends Message {}    // Notifies about change
```

#### Domain Event Structure

```php
// src/Domain/Plays/Events/PlayCreated.php
final readonly class PlayCreated implements Event
{
    public function __construct(
        public PlayId $playId,
        public GameId $gameId,
        public DateTimeImmutable $occurredAt,
    ) {}
}
```

#### Event Handler

```php
// src/Application/Handlers/Plays/OnPlayCreated/Handler.php
final readonly class Handler implements EventHandler
{
    public function __construct(
        private StatsUpdater $statsUpdater,
    ) {}

    public function handle(PlayCreated $event): void
    {
        $this->statsUpdater->incrementPlayCount($event->gameId);
    }
}
```

#### Outbox Table (Beta)

```sql
CREATE TABLE outbox
(
    id             UUID PRIMARY KEY,
    aggregate_type VARCHAR(255) NOT NULL,
    aggregate_id   UUID         NOT NULL,
    event_type     VARCHAR(255) NOT NULL,
    payload        JSONB        NOT NULL,
    created_at     TIMESTAMP    NOT NULL,
    processed_at   TIMESTAMP    NULL,
    attempts       INT DEFAULT 0
);
```

#### When to Apply What

| Task                                 | Solution                     | Phase     |
|--------------------------------------|------------------------------|-----------|
| Logic decomposition within module    | Domain events via MessageBus | MVP       |
| Reliable async communication         | Outbox + Jobs                | Expansion |
| Full audit with state reconstruction | Event Sourcing               | Scaling*  |

*Only for aggregates with explicit need.
